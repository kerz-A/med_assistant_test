# Ассистент заполнения медицинского протокола в реальном времени

Тестовая реализация системы реального времени для автоматического заполнения медицинского протокола во время приёма врача. Врач ведёт приём как обычно — система слушает аудиопоток через WebSocket, транскрибирует речь, определяет кто говорит (врач или пациент), извлекает медицинские данные и заполняет структурированный протокол на лету.

> **Важно:** Это тестовая/прототипная реализация. В production при наличии достаточных GPU-ресурсов все ML-модели (ASR, VAD, Speaker ID) должны работать на GPU для эффекта заполнения полей в реальном времени. Текущая сборка поддерживает режимы CPU и GPU.

## Быстрый старт

```bash
cp .env.example .env
# В .env указать: LLM_PROVIDER=gigachat и GIGACHAT_AUTH_KEY=...
```

### Развёртывание на GPU (рекомендуется)

```bash
docker compose -f docker-compose.yml -f docker-compose.gpu.yml up --build --no-deps backend frontend
```

Все ML-модели работают на GPU — ASR обрабатывает сегменты за <1с, обеспечивая обновление полей в реальном времени. Требуется NVIDIA драйвер с поддержкой CUDA 12.1.

### Развёртывание на CPU (достаточно для тестирования)

```bash
docker compose up --build --no-deps backend frontend
```

ASR работает в ~3-5 раз медленнее реального времени. Установите `WHISPER_MODEL=small` в `.env` для ускорения с небольшой потерей качества.

Фронтенд: http://localhost:3000 | Health check: http://localhost:8000/api/health

## 4 стадии приёма

### 1. Калибровка

Врач просит пациента представиться: _"Назовите ФИО, возраст и пол"_. Система записывает 2 реплики (врач + пациент), по ним:

- **Silero VAD** сегментирует речь, отделяя паузы от речевых фрагментов
- **Whisper** транскрибирует каждый сегмент
- **ECAPA-TDNN** вычисляет голосовые эмбеддинги — первый спикер = врач, второй = пациент
- **LLM** извлекает из ответа пациента: ФИО, возраст, пол

После калибровки система знает голос врача и голос пациента и может классифицировать все последующие реплики.

### 2. Запись (основной приём)

Аудио поступает через WebSocket чанками по 100мс. Обработка:

1. **VAD** непрерывно анализирует поток — при обнаружении речи (>500мс) формирует сегмент
2. Каждый сегмент параллельно обрабатывается (concurrency=2 на GPU, 1 на CPU):
   - **Whisper** транскрибирует речь (RTF ~0.6-1.0 на GPU)
   - **ECAPA-TDNN** сравнивает эмбеддинг с эталонами врача/пациента (cosine similarity)
3. Результат добавляется в транскрипт и pending-очередь
4. Каждые **5 utterances** или **20 секунд** — батчевый вызов LLM для извлечения данных в протокол

Протокол заполняется инкрементально: жалобы, анамнез, аллергии, лекарства, витальные показатели.

### 3. Остановка записи

Врач нажимает "Стоп". Система:

- Flush VAD буфера (последний незавершённый сегмент)
- Дообработка оставшихся в очереди сегментов (ожидание до 300с)
- Финальная extraction pending utterances, если остались необработанные

### 4. Финализация (заключение)

LLM получает протокол и контекст диалога, генерирует:

- Предварительный диагноз по МКБ-10
- План лечения и обследований
- Рекомендации пациенту простым языком + RED FLAGS

Для длинных сессий (>50 реплик) используется автоматическая суммаризация реплик пациента — транскрипт сжимается с ~27 КБ до ~5-7 КБ, сохраняя все медицински значимые данные.

## Архитектура

### Pipeline обработки аудио

```
Микрофон → WebSocket (100мс чанки, 16kHz mono)
    → Silero VAD (~2мс на чанк)
        → при обнаружении речи (>500мс):
            ├─ Whisper medium (RTF 0.6-1.0 GPU / 3-5x CPU)  ─┐
            └─ ECAPA-TDNN (~20мс GPU / ~150мс CPU)           ─┤
                                                               ↓
                                                 Utterance (text + speaker)
                                                               ↓
                                         Pending Queue → Batch (5 utt / 20с)
                                                               ↓
                                                      LLM Extraction (облако)
                                                               ↓
                                                    Medical Protocol (JSON)
```

### Компоненты

| Компонент | Технология | Назначение | GPU | CPU |
|-----------|-----------|------------|-----|-----|
| VAD | Silero VAD v5 (~5 МБ) | Детекция речи/тишины в потоке | ~2мс/чанк | ~5мс/чанк |
| ASR | faster-whisper medium (~1.5 ГБ) | Транскрипция русской речи, beam_size=5 | ~0.5-1с/сегмент | ~3-5с/сегмент |
| Speaker ID | ECAPA-TDNN SpeechBrain (~120 МБ) | Cosine similarity голосовых эмбеддингов | ~20мс | ~100-200мс |
| LLM (облако) | GigaChat-2-Max | Extraction + суммаризация + финализация | 0 (API) | 0 (API) |
| LLM (локальный) | Ollama (qwen2.5:14b+) | Полностью автономная работа без интернета | ~10 ГБ VRAM | медленно |

**Минимальные требования GPU**: ~2 ГБ VRAM (Whisper medium + VAD + ECAPA с int8_float32). Рекомендуется 4+ ГБ VRAM для float16 и запаса при пиковой нагрузке.

## Выбор LLM-провайдера

### Вариант A: GigaChat (облачный, рекомендовано)

- **Оплата из РФ** — российские карты (Mir, SberPay), юрлица, ИП
- **Нет RPM/TPM лимитов** — ограничение только по concurrency (1 поток на free tier)
- **Оптимизирован для русского** — понимает медицинскую терминологию, МКБ-10

```env
LLM_PROVIDER=gigachat
GIGACHAT_AUTH_KEY=ваш_authorization_key_из_developers_sber_ru
GIGACHAT_MODEL=GigaChat-2-Max
```

Получение ключа: [developers.sber.ru](https://developers.sber.ru) → Создать проект → Подключить GigaChat API → Скопировать Authorization Key. Freemium: 1M токенов/год бесплатно (~100 сессий).

**Особенности**: OAuth токен (30 мин TTL) — автоматическое обновление при 401, self-signed SSL обрабатывается в коде.

### Вариант B: Ollama (локальный, без интернета)

Полностью автономная работа на собственном GPU. Не требует API-ключей и интернета.

```env
LLM_PROVIDER=ollama
OLLAMA_BASE_URL=http://ollama:11434
OLLAMA_MODEL=qwen2.5:3b   # по умолчанию; рекомендуется 14b при наличии GPU
```

**Модели Ollama:**

| Модель | VRAM | Качество | Скорость |
|--------|------|----------|----------|
| `qwen2.5:3b` | ~2 ГБ | Базовое (дефолт, не знает МКБ-10) | ~1-3с/запрос |
| `qwen2.5:7b` | ~5 ГБ | Среднее | ~3-8с/запрос |
| `qwen2.5:14b` | ~10 ГБ | Хорошее (рекомендовано при 12+ ГБ VRAM) | ~5-15с/запрос |

**Требования**: GPU с достаточным VRAM для выбранной модели + ~2 ГБ для Whisper. Модель автоматически скачивается при первом запуске.

> **Важно**: при локальном режиме Ollama — это единственный LLM, fallback отсутствует. При облачном режиме (GigaChat) Ollama автоматически используется как fallback при сбоях API.

### Ранее протестированные альтернативы

| Провайдер | Конфигурация | Почему не подошёл |
|-----------|-------------|-------------------|
| **Groq** (free) | `LLM_PROVIDER=groq` | 6,000 TPM — 3-й вызов за сессию уже получает 429 |
| **OpenRouter** (free) | `LLM_PROVIDER=openrouter` | 50 запросов/день — 2 сценария исчерпывают лимит |
| **DeepSeek** | `LLM_PROVIDER=deepseek` | Хорошее качество, $0.28/M tok, но оплата из РФ затруднена |
| **OpenAI** | `LLM_PROVIDER=openai` | GPT-4o-mini, наивысшее качество, но дороже и требует зарубежную карту |

> Код поддерживает эти провайдеры (конфигурация в `config.py`), но они не рекомендуются для production из-за ограничений выше.

## Отказоустойчивость LLM

Система гарантирует обработку каждого utterance без потери данных:

1. **Retry с cooldown внутри Lock**: при 429 — экспоненциальный backoff (до 8с), cooldown timestamp виден всем concurrent вызовам через `asyncio.Lock`
2. **GigaChat 401 auto-refresh**: при протухшем токене — автоматическое обновление OAuth и повтор запроса (один раз)
3. **Confirm only on success**: удаление из pending-очереди только после успешного LLM-ответа. При ошибке utterances остаются и обрабатываются в следующем цикле
4. **Ollama fallback**: при облачном LLM — автоматический вызов к локальному Ollama при исчерпании retry. Качество ниже, но данные не теряются
5. **Финализация компенсирует**: даже если инкрементальная extraction пропустила данные, финализация видит весь контекст и компенсирует пробелы

## Поддержка длительных сессий

Сессии свыше ~50 реплик (~15+ минут) используют двухэтапную финализацию:

1. **Суммаризация** — из транскрипта извлекаются только реплики пациента, LLM сжимает их в 15-20 предложений с сохранением медицинских формулировок
2. **Финализация** — протокол + суммаризация (вместо полного транскрипта) → ~5-7 КБ вместо ~27 КБ

Это предотвращает переполнение контекста LLM и деградацию качества на длинных приёмах.

## Система помощи принятию врачебного решения (CDS)

После финализации приёма система автоматически оценивает качество консультации по двум блокам.

### Критерии качества (0/1, 11 критериев)

Каждый критерий оценивается по шкале:
- **0** — тема не затронута вообще
- **1** — тема затронута поверхностно / частично
- **2** — тема раскрыта полностью и подробно

| Критерий | Что оценивается |
|----------|----------------|
| Приветствие и контакт | Представление, установление доверительного контакта |
| Структура разговора | Логичная последовательность: жалобы → анамнез → осмотр → план |
| Выявление потребностей | Ожидания и опасения пациента |
| Текущие жалобы | Локализация, характер, интенсивность (ВАШ), провоцирующие факторы |
| Анамнез заболевания | Начало, динамика, предпринятое лечение и его эффект |
| Общий мед. анамнез | Хронические болезни, операции, госпитализации |
| Лекарственный анамнез | Препараты, дозы, кратность, длительность приёма |
| Семейный анамнез | Заболевания у родителей, братьев/сестёр |
| Профилактика и риски | Курение, алкоголь, питание, сон, физ. активность, стресс |
| Планирование лечения | Обследования, назначения с обоснованием, консультации |
| Заключение визита | Итоги, рекомендации, red flags, контрольный визит |

**Итоговая оценка**: сумма баллов / 22 × 5 = оценка по 5-балльной шкале.

### Аналитика диалога (10 метрик)

Бинарные метрики (0/1):

| Метрика | Шкала | Описание |
|---------|-------|----------|
| Эмпатия врача | 0/1 | Проявление сочувствия к пациенту |
| Врач не перебивал пациента | 0/1 | Давал пациенту договорить, выдерживал паузы |
| Пациент задавал вопросы | 0/1 | Активность пациента в диалоге |
| Медицинский жаргон | 0/1 | Использование терминов без пояснений |
| Подтверждение понимания | 0/1 | Врач уточнял понимание пациента |
| Обсуждение образа жизни | 0/1 | Питание, сон, стресс, физ. активность |
| Обсуждение аллергий | 0/1 | Были ли выяснены аллергии |
| Совместное принятие решений | 0/1 | Участие пациента в выборе лечения |
| Оценка комплаентности | 0/1 | Оценка приверженности лечению |
| Комфортный темп | 0/1 | Врач не торопился, давал время на ответы |

### CDS в автотестах

- Детали по каждому сценарию: оценка, количество заполненных критериев

## Известные ограничения

- **Speaker ID на одном поле**: при одинаковых голосах (два мужчины / две женщины) точность SA падает с ~82% до ~75% — ECAPA-TDNN эмбеддинги слишком похожи
- **GigaChat double braces**: GigaChat иногда оборачивает JSON в `{{ }}` — парсер автоматически исправляет
- **GigaChat BP формат**: возвращает давление как "150/95 мм рт. ст." — парсер извлекает числа через regex
- **Ollama + Whisper на одном GPU**: конкурируют за VRAM. Рекомендуется GPU с 12+ ГБ VRAM для qwen2.5:14b + Whisper medium
- **CPU-режим**: ASR в 3-5 раз медленнее реального времени, эффект "заполнения на лету" пропадает. Для тестирования работоспособности — достаточно

## Переменные окружения

Полный список см. в `.env.example`. Основные настройки:

```env
# LLM
LLM_PROVIDER=gigachat          # gigachat | ollama
GIGACHAT_AUTH_KEY=...           # обязателен для gigachat
OLLAMA_MODEL=qwen2.5:14b       # для ollama

# Whisper ASR
WHISPER_MODEL=medium            # tiny | base | small | medium | large-v3
WHISPER_DEVICE=auto             # auto | cpu | cuda
WHISPER_COMPUTE_TYPE=float16    # float16 (GPU) | int8 (CPU) | int8_float32
```
