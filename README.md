# Aiston TT v4 — Realtime заполнение протокола медицинского осмотра

Система реального времени для автоматического заполнения медицинского протокола во время приёма врача. Врач ведёт приём как обычно — система слушает аудиопоток через WebSocket, транскрибирует речь, определяет кто говорит (врач или пациент), извлекает медицинские данные и заполняет структурированный протокол на лету.

## Быстрый старт

```bash
cp .env.example .env
# В .env указать: LLM_PROVIDER=gigachat и GIGACHAT_AUTH_KEY=...

# Запуск с GPU (рекомендовано):
docker compose -f docker-compose.yml -f docker-compose.gpu.yml up --build

# Запуск без GPU (CPU, медленнее ~3-5x):
docker compose up --build
```

Frontend: http://localhost:3000 | Backend API: http://localhost:8000/api/health

## 4 стадии приёма

### 1. Калибровка

Врач просит пациента представиться: _"Назовите ФИО, возраст и пол"_. Система записывает 2 реплики (врач + пациент), по ним:

- **Silero VAD** сегментирует речь, отделяя паузы от речевых фрагментов
- **Whisper** транскрибирует каждый сегмент
- **ECAPA-TDNN** вычисляет голосовые эмбеддинги — первый спикер = врач, второй = пациент
- **LLM** извлекает из ответа пациента: ФИО, возраст, пол

После калибровки система знает голос врача и голос пациента, и может классифицировать все последующие реплики.

### 2. Запись (основной приём)

Аудио поступает через WebSocket чанками по 100мс. Обработка:

1. **VAD** непрерывно анализирует поток — при обнаружении речи (>500мс) формирует сегмент
2. Каждый сегмент параллельно обрабатывается (concurrency=2 на GPU):
   - **Whisper** транскрибирует речь (RTF ~0.6-1.0 на GPU)
   - **ECAPA-TDNN** сравнивает эмбеддинг с эталонами врача/пациента (cosine similarity)
3. Результат добавляется в транскрипт и pending-очередь
4. Каждые **5 utterances** или **20 секунд** — батчевый вызов LLM для извлечения данных в протокол

Протокол заполняется инкрементально: жалобы, анамнез, аллергии, лекарства, витальные показатели.

### 3. Остановка записи

Врач нажимает "Стоп". Система:

- Flush VAD буфера (последний незавершённый сегмент)
- Дообработка оставшихся в очереди сегментов (ожидание до 60с)
- Финальная extraction pending utterances, если остались необработанные

### 4. Финализация (заключение)

LLM получает **полный транскрипт** диалога и текущий протокол, генерирует:

- Предварительный диагноз по МКБ-10
- План лечения и обследований
- Рекомендации пациенту

Это самый критичный вызов — он видит весь контекст, поэтому даже если инкрементальная extraction что-то пропустила, финализация компенсирует.

## Выбор LLM-провайдера

### GigaChat-2-Max (рекомендовано)

Основной провайдер. Причины выбора:

- **Оплата из РФ** — российские карты (Mir, SberPay), юрлица, ИП
- **Нет RPM/TPM лимитов** — только ограничение по concurrency (1 поток на free tier, совместимо с архитектурой приложения через `asyncio.Lock`)
- **Отличный русский язык** — обучена на русскоязычных данных
- **Медицинская диагностика** — 62.5% точность МКБ-10 на GigaChat-2-Pro, неверные диагнозы в правильной категории (J22 вместо J20, L20 вместо L23)
- **Приемлемая скорость** — 2-12 секунд на вызов

Настройка:

```bash
LLM_PROVIDER=gigachat
GIGACHAT_AUTH_KEY=ваш_authorization_key_из_developers_sber_ru
GIGACHAT_MODEL=GigaChat-2-Max
```

Получение ключа: [developers.sber.ru](https://developers.sber.ru) → Создать проект → Подключить GigaChat API → Скопировать Authorization Key. Freemium: 1M токенов/год бесплатно (~100 сессий). Пакет GigaChat Pro: 3M токенов за 1500 руб/год.

**Особенности интеграции**: GigaChat использует OAuth токен (30 мин TTL, автоматическое обновление) и self-signed SSL сертификат (обрабатывается в коде).

### Протестированные альтернативы и почему они не подошли

| Провайдер | Проблема | Детали |
|-----------|----------|--------|
| **Groq** (free) | Жёсткий rate limit | 6,000 TPM — при 7 вызовах за сессию (~1,500 tok каждый) третий вызов уже получает 429. Все extraction/finalization падали на Ollama fallback |
| **OpenRouter** (free) | 50 запросов в день | На free tier без покупки кредитов — 50 RPD. Два сценария = 14 вызовов + retry = лимит исчерпан |
| **DeepSeek** | Оплата из РФ затруднена | Нет фиксированных rate limits, отличная цена ($0.28/M tok), но требует зарубежную карту или посредника. Поддержка добавлена в код |
| **Ollama** (qwen2.5:3b) | Низкое качество + медленно | DA=0% (модель 3B не знает МКБ-10). На shared GPU (GTX 1660) — 30-120 секунд на запрос из-за конкуренции с Whisper за VRAM |

## Архитектура

### Pipeline обработки аудио

```
Микрофон → WebSocket (100мс чанки, 16kHz mono)
    → Silero VAD (GPU, ~1мс на чанк)
        → при обнаружении речи (>500мс):
            ├─ Whisper medium (GPU, RTF 0.6-1.0)  ─┐
            └─ ECAPA-TDNN (GPU, ~50мс)            ─┤
                                                    ↓
                                              Utterance (text + speaker)
                                                    ↓
                                    Pending Queue → Batch (5 utt / 20s)
                                                    ↓
                                              LLM Extraction
                                                    ↓
                                           Medical Protocol (JSON)
```

### Компоненты

| Компонент | Технология | Назначение | VRAM |
|-----------|-----------|------------|------|
| VAD | Silero VAD v5 | Детекция речи/тишины в потоке | ~50 MB |
| ASR | faster-whisper medium | Транскрипция ru/en, beam_size=5 | ~2.5 GB |
| Speaker ID | ECAPA-TDNN (SpeechBrain) | Cosine similarity эмбеддингов | ~500 MB |
| LLM (облако) | GigaChat-2-Max | Extraction + финализация | 0 (API) |
| LLM (fallback) | Ollama qwen2.5:3b | Резервный при отказе облака | ~2 GB |

**Минимальные требования GPU**: 4 GB VRAM (Whisper medium + VAD + ECAPA). С Ollama fallback: 6 GB.

### Отказоустойчивость LLM (zero data loss)

Система гарантирует обработку каждого utterance:

1. **Retry с cooldown внутри Lock**: при 429 — экспоненциальный backoff (до 8с), cooldown timestamp виден всем concurrent вызовам через `asyncio.Lock`. Предотвращает гонку, когда два вызова одновременно получают 429
2. **Confirm only on success**: `confirm_extraction()` (удаление из pending очереди) вызывается только после успешного LLM-ответа. При ошибке utterances остаются в очереди и обрабатываются в следующем цикле
3. **Ollama fallback**: при исчерпании retry облачного провайдера — автоматический вызов к локальному Ollama. Качество ниже, но данные не теряются
4. **Финализация по полному транскрипту**: даже если инкрементальная extraction пропустила данные, финализация видит все utterances и компенсирует пробелы

### Известные ограничения

- **Speaker ID на одном поле**: при одинаковых голосах (два мужчины или две женщины) точность SA падает с 88.6% до 75% — ECAPA-TDNN эмбеддинги слишком похожи
- **GigaChat double braces**: GigaChat иногда оборачивает JSON в `{{ }}` — парсер автоматически исправляет, но при сложной вложенности возможны edge cases
- **GigaChat BP формат**: возвращает давление как "150/95 мм рт. ст." вместо "150/95" — парсер извлекает числа через regex
- **Ollama на shared GPU**: на GTX 1660 (6 GB) Ollama конкурирует с Whisper за VRAM, время ответа 30-120 секунд. На выделенном GPU или CPU-only — стабильнее
- **WebSocket keepalive**: при длительном аудио (>2 мин) тестовый стример может disconnect. В production с живым микрофоном проблема не воспроизводится

### Метрики качества (GigaChat-2-Pro, 8 завершённых сценариев)

| Метрика | Значение | Что измеряет |
|---------|----------|-------------|
| **TC** | **100%** | Полнота транскрипции — все реплики захвачены без потерь |
